## 🧠 DDD(Domain-Driven Design) 개념

- **에릭 에반스(Eric Evans)** 가 2003년에 소개한 소프트웨어 설계 접근법
    
- **복잡한 비즈니스 로직**을 소프트웨어에 효율적으로 반영하기 위해 고안됨
    
- 핵심: **비즈니스 도메인(domain)에 집중하고, 모델을 중심으로 설계**
    

---

### 🔹 핵심 개념

| 개념                      | 설명                            |
| ----------------------- | ----------------------------- |
| **Domain (도메인)**        | 해결하고자 하는 비즈니스 문제 영역           |
| **Entity (엔티티)**        | 고유한 식별자를 가진 객체, 상태 변화 가능      |
| **Value Object (값 객체)** | 식별자가 없고, 값 자체로 의미가 있는 객체      |
| **Aggregate (애그리거트)**   | 관련 객체 그룹을 하나로 묶고 일관성 유지       |
| **Repository (리포지토리)**  | 도메인 객체를 저장/조회하는 계층            |
| **Service (서비스)**       | 엔티티나 값 객체가 담당하지 않는 비즈니스 로직 구현 |
| **Bounded Context**     | 용어, 규칙, 모델이 명확히 정의된 독립적인 영역   |
