좋아요 👍 RESTful API를 설계할 때는 **일관성, 단순성, 예측 가능성**이 핵심 원칙입니다.  
아래는 **RESTful API 구성 권고안 (Best Practices)**입니다 — 실제 프로젝트 설계 시 기준 가이드로 삼기 좋습니다.

---

## 🌐 1. 기본 설계 원칙

| 원칙                   | 설명                                                                                |
| -------------------- | --------------------------------------------------------------------------------- |
| **리소스 중심 설계**        | API는 “행동(verbs)”보다 “리소스(nouns)” 중심으로 설계해야 합니다. 예: `/users` (O) vs `/getUsers` (X) |
| **HTTP 메서드 사용**      | CRUD에 맞는 HTTP 메서드를 명확히 구분해야 합니다.                                                  |
| **Stateless (무상태성)** | 서버는 클라이언트의 상태를 보존하지 않습니다. 모든 요청은 완결된 정보를 포함해야 합니다.                                |
| **일관성 있는 URL 구조**    | 동일한 패턴과 규칙을 URL 전반에 일관되게 적용합니다.                                                   |
| **표준 HTTP 상태 코드 사용** | 성공, 실패, 예외 상황을 명확히 표현합니다.                                                         |

---

## 🚏 2. HTTP 메서드 사용 규칙

| 동작        | 메서드      | 예시                   |     |
| --------- | -------- | -------------------- | --- |
| 리소스 목록 조회 | `GET`    | `GET /users`         |     |
| 단일 리소스 조회 | `GET`    | `GET /users/{id}`    |     |
| 리소스 생성    | `POST`   | `POST /users`        |     |
| 리소스 전체 수정 | `PUT`    | `PUT /users/{id}`    |     |
| 리소스 일부 수정 | `PATCH`  | `PATCH /users/{id}`  |     |
| 리소스 삭제    | `DELETE` | `DELETE /users/{id}` |     |

---

## 🧭 3. URL (엔드포인트) 설계 원칙

✅ **명사 사용 (복수형)**

`/users           → 사용자 목록   /users/123       → 특정 사용자   /users/123/posts → 특정 사용자의 게시물`

✅ **중첩 리소스는 계층적으로 표현**

`/companies/77/employees/3`

✅ **쿼리 파라미터는 필터, 정렬, 페이징에 사용**

`GET /products?category=phone&sort=price&order=asc&page=2&limit=20`

✅ **하이픈(-) 사용, 언더스코어(_) 지양**

`/users-profile (O) /users_profile (X)`

---

## 🧩 4. 응답(Response) 형식

✅ **JSON 형식 권장**

`{   "data": {     "id": 1,     "name": "Kim",     "email": "kim@example.com"   },   "meta": {     "timestamp": "2025-10-24T13:45:00Z"   } }`

✅ **일관된 에러 응답 구조**

`{   "error": {     "code": "USER_NOT_FOUND",     "message": "User not found",     "status": 404   } }`

---

## 🚨 5. HTTP 상태 코드 가이드

| 상태     | 코드                        | 의미              |
| ------ | ------------------------- | --------------- |
| 성공     | 200 OK                    | 요청 성공           |
| 생성     | 201 Created               | 새 리소스 생성됨       |
| 없음     | 204 No Content            | 성공했지만 반환 데이터 없음 |
| 잘못된 요청 | 400 Bad Request           | 클라이언트 요청 오류     |
| 인증 실패  | 401 Unauthorized          | 인증 필요           |
| 권한 없음  | 403 Forbidden             | 접근 권한 없음        |
| 리소스 없음 | 404 Not Found             | 해당 리소스 없음       |
| 서버 오류  | 500 Internal Server Error | 서버 내부 오류        |

---

## 🛡️ 6. 보안 및 버저닝

✅ **API 버전은 URL 또는 Header로 관리**

`GET /v1/users # 또는 GET /users Header: Accept-Version: v1`

✅ **HTTPS만 사용**

> 민감한 정보 보호를 위해 HTTP는 금지.

✅ **인증 방식**

- JWT (JSON Web Token)
    
- OAuth 2.0
    
- API Key (단순한 서비스용)
    

---

## ⚙️ 7. 기타 권장 사항

- ✅ CORS 정책을 명확히 설정
    
- ✅ Rate Limiting (요청 제한)
    
- ✅ API 문서는 Swagger/OpenAPI로 자동화
    
- ✅ Pagination, Sorting, Filtering 일관된 구조로 제공
    
- ✅ HATEOAS는 필요 시 링크 기반 탐색 제공
