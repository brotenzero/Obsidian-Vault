# ✅ **@Component가 먼저 적용되고, 그 다음에 @Transactional이 적용된다.**

왜 그런지 구조를 단계별로 설명해줄게.

---

# 🔍 1. @Component는 “빈 등록” 단계에서 실행됨

@Component, @Service, @Repository 등은  
**스프링 컨테이너가 시작될 때 객체를 스캔하고 빈으로 등록하는 과정에서 작동**함.

즉, 스프링 컨테이너 초기화 단계에서 적용됨.

✔ 빈 생성  
✔ 의존성 주입  
✔ 초기화 완료

이 과정이 끝나야 스프링 빈이 실제로 "사용 가능" 상태가 됨.

---

# 🔍 2. @Transactional은 “메소드 실행 시점”에 동작

@Transactional은 빈이 만들어진 후에,  
**AOP 프록시(proxy)**가 해당 빈을 감싸는 방식으로 적용됨.

따라서 @Transactional은 아래 시점에 작동함:

- 메소드가 호출될 때
    
- 프록시가 가로채서
    
- 트랜잭션을 시작/커밋/롤백
    

즉, 빈 등록이 끝난 다음 “실행 시점”에 동작하는 기능이야.

# 1️⃣ **Spring Container (스프링 컨테이너)**

### 정의

- **스프링이 관리하는 객체(Bean)를 담아두는 공간**
    
- 공식 명칭: **ApplicationContext**
    
- IoC(Inversion of Control)와 DI(Dependency Injection)를 지원
    

### 특징

- 애플리케이션 실행 시 메모리에서 생성
    
- 객체의 생성, 의존성 주입, 라이프사이클 관리
    
- 클래스 단위, JVM 내부에서만 존재
    
- 예: `@Component`, `@Service`, `@Repository` 빈들이 Spring Container 안에 저장됨
    

### 요약

> **“애플리케이션 객체 관리 공간”**  
> 단지 **메모리상의 객체 저장소**이고, OS나 네트워크와는 무관함

---

# 2️⃣ **Docker Container (도커 컨테이너)**

### 정의

- 애플리케이션과 그 환경을 **패키징해서 격리된 공간에서 실행하는 단위**
    
- OS 수준에서 프로세스를 격리
    

### 특징

- OS 커널을 공유하지만, 독립된 환경 제공
    
- 필요한 라이브러리, 실행환경, 설정까지 포함
    
- 네트워크, 파일시스템도 격리됨
    
- 예: 하나의 컨테이너에서 Spring Boot 앱이 실행되고, 다른 컨테이너에서 MySQL DB가 실행
    

### 요약

> **“애플리케이션 실행 환경 단위”**  
> OS 수준에서 격리되어 실행됨, 메모리와 프로세스를 포함
