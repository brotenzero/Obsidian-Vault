## ✅ **스택(Stack)**

👉 **LIFO(Last In, First Out)** 구조  
→ 마지막에 들어간 데이터가 가장 먼저 나온다.

### 예시

- 책을 위로 쌓아 올린 더미
    
- 웹 브라우저 뒤로 가기 기능
    

### 주요 연산

- **push**: 데이터를 스택의 top(위)에 추가
    
- **pop**: top(위)에서 데이터 꺼내기
    
- **peek**: top 값 확인
    

---

## ✅ **큐(Queue)**

👉 **FIFO(First In, First Out)** 구조  
→ 먼저 들어간 데이터가 먼저 나온다.

### 예시

- 줄 서기(선착순)
    
- 프린터 출력 대기열
    

### 주요 연산

- **enqueue**: 데이터를 큐의 뒤(Rear)에 추가
    
- **dequeue**: 큐의 앞(Front)에서 데이터 꺼내기
    
- **peek**: front 값 확인

# 1. 아토믹 디자인(Atomic Design)이란?

- Brad Frost가 제안한 UI 설계 방법론
    
- **UI를 작은 단위부터 큰 단위로 조합**하여 구성
    
- HTML, CSS, React 같은 컴포넌트 기반 개발과 잘 맞음
    

**핵심 아이디어:**

> “UI를 재사용 가능한 작은 조각(원자)으로 나누고, 이를 조합해 큰 화면을 만든다”

---

# ✅ 2. 아토믹 디자인 구조

아래처럼 **5단계 구조**로 나눔:

|단계|설명|React 예시|
|---|---|---|
|**Atom (원자)**|UI의 가장 작은 단위, 재사용 가능|버튼(Button), 입력(Input), 라벨(Label)|
|**Molecule (분자)**|Atom 조합, 간단한 UI 단위|검색창(SearchBox = Input + Button)|
|**Organism (유기체)**|Molecule + Atom 조합, 복잡한 UI 블록|헤더(Header = 로고 + 네비게이션 + 검색창)|
|**Template (템플릿)**|Organism을 배치, 구조 정의|페이지 레이아웃(헤더 + 본문 + 사이드바)|
|**Page (페이지)**|실제 데이터 적용, 최종 UI|홈 페이지(HomePage), 제품 상세 페이지(ProductPage)|

---

# ✅ 3. React에서 적용 방법

### 1) Atom 예시

`// atoms/Button.jsx export default function Button({ children, onClick }) {   return <button onClick={onClick}>{children}</button>; }`

### 2) Molecule 예시

`// molecules/SearchBox.jsx import Button from '../atoms/Button'; import Input from '../atoms/Input';  export default function SearchBox({ value, onChange, onSearch }) {   return (     <div>       <Input value={value} onChange={onChange} />       <Button onClick={onSearch}>검색</Button>     </div>   ); }`

### 3) Organism 예시

`// organisms/Header.jsx import Logo from '../atoms/Logo'; import NavMenu from '../molecules/NavMenu'; import SearchBox from '../molecules/SearchBox';  export default function Header() {   return (     <header>       <Logo />       <NavMenu />       <SearchBox />     </header>   ); }`

# ✅ 4. 아토믹 패턴의 장점

1. **재사용성 극대화**
    
    - Atom 단위를 여러 Molecule/Organism에서 재사용 가능
        
2. **유지보수 용이**
    
    - 작은 단위 수정 → 전체 UI에 반영
        
3. **일관성 있는 디자인**
    
    - 디자인 시스템과 쉽게 연계 가능
        
4. **테스트 용이**
    
    - 작은 컴포넌트를 단위 테스트 가능

# ✅ 1. SPA가 무엇인가?

전통적인 웹사이트는 페이지를 이동할 때마다:

- 서버에서 새로운 HTML 다운로드
    
- 전체 화면 다시 렌더링
    

하지만 **SPA는 오직 한 번만 HTML을 로드**하고,  
이후에는 **JavaScript가 화면만 부분적으로 바꿔줍니다.**

즉:

- **페이지 전환은 일어나지만**
    
- 실제로 **브라우저는 새 페이지를 요청하지 않음**
    
- 라우팅도 JavaScript가 담당 (React Router 등)

# ✅ 1. UML에서 “사용(use) 관계”란?

> **어떤 요소가 다른 요소의 기능을 일시적으로 사용한다는 것을 의미하는 가장 약한 의존 관계**

즉,

- 클래스 A가 클래스 B의 기능을 _잠깐_ 필요로 할 때
    
- A → B로 “사용 관계”를 설정합니다.
    

---

# ✅ 2. 예시

`A ----uses----> B`

- A는 B의 기능을 “사용”하지만
    
- A가 B를 소유하는 것도 아니고
    
- A의 생명주기가 B에 종속되는 것도 아님
    
- 그냥 “필요할 때 호출하는 정도”
    

---

# ✅ 3. Java 코드 예시

`class Printer {     void printDocument() { } }  class ReportService {     void generate() {         Printer printer = new Printer(); // 사용 관계         printer.printDocument();     } }`

- ReportService는 Printer를 **일시적으로 사용**합니다.
    
- Printer를 필드로 갖지 않아도 관계 성립
    

---

# ✅ 4. 어떤 관계들과 비교할까?

|관계|의미|강도|
|---|---|---|
|**사용 관계(use)**|잠깐 기능만 사용|가장 약함|
|**연관(association)**|두 객체가 지속적으로 연결|중간|
|**집합(aggregation)**|전체–부분 관계(약함)|약중|
|**합성(composition)**|전체–부분 관계(강함, 생명주기 공유)|강함|
|**상속(generalization)**|extends / 상속 구조|매우 강함|

**사용 관계는 가장 약한 관계**입니다.

---

# ✅ 5. 정리

- “**use 관계 = 사용 관계**”
    
- UML에서 가장 약한 의존 관계
    
- 한 객체가 다른 객체의 기능을 잠깐 호출할 때 표현
    
- Java, JS, React 모두 같은 의미로 이해 가능
